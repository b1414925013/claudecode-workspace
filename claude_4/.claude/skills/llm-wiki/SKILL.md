# LLM Wiki 知识库管理系统

## 目标

维护一个**可积累、可链接、可演进**的 Markdown / Obsidian 知识库。在导入资料或回答问题时，不仅仅是生成摘要，而是将新知识编译到现有页面中：更新实体、概念、综合判断、对比和查询归档，使知识库通过持续使用变得更有价值。

## 默认三层结构

```text
source/          # 原始材料层：事实性来源，默认只读
wiki/            # 知识编译层：由 LLM 维护的结构化知识库
schema/SKILL.md  # 约定层：本技能/规范文件
```

### 知识库根目录

`Knowledge/` 根目录只应包含少量结构性入口点和工具文件。不要将普通的知识页面放在这里。

根目录推荐内容：

```text
Knowledge/
  source/
  wiki/
  schema/
  verify_wiki.py   # 可选兼容包装，委托给 schema/scripts/verify_wiki.py
  README.md        # 可选：最小化入口
```

根目录**不应包含**：

- 单源摘要；
- 概念页；
- 综合页；
- 主题图谱；
- 临时整理报告。

如果需要根入口点，创建最小化的 `Knowledge/README.md`，只链接到：

- `[[Knowledge/wiki/README]]`
- `[[Knowledge/wiki/index]]`
- `[[Knowledge/wiki/syntheses/知识库主题图谱]]`
- `[[Knowledge/wiki/log]]`
- `[[Knowledge/source]]`

**原则**：根目录是"入口点"；`wiki/` 是"知识"。

### source/ 原始材料层

- 原始材料默认为**只读**。不要重写其正文内容。
- 保留原始标题、作者、URL、日期、剪辑元数据、图片引用等来源信息。
- 如果需要对原始材料重命名或补充元数据，先向用户确认。

### wiki/ 知识编译层

```text
wiki/
  index.md      # 主索引（保留英文文件名）
  log.md        # 操作日志（保留英文文件名）
  README.md     # 知识库文档（保留英文文件名）
  sources/      # 来源页
  entities/     # 实体页
  concepts/     # 概念页
  syntheses/    # 综合页
  comparisons/  # 对比页
  queries/      # 查询归档
```

约定规范：

- 除约定的 `index.md`、`log.md`、`README.md`、`schema/SKILL.md` 外，目录和文件名优先使用可读的中文或英文命名
- 术语专有名词可保留英文或使用中英混合命名，例如 `AI Agent.md`、`OpenClaw.md`、`Claude Code.md`、`Harness Engineering.md`
- 目录名使用英文复数形式，如 `sources/`、`concepts/`、`entities/`
- 文件名应可读、可点击、长期可维护，避免机器风格的前缀如 `source-*`、`concept-*`、`synthesis-*`

## 页面类型

### 来源页：`wiki/sources/`

来源页用于概括和定位单个来源。应包含：

- **来源信息**：原始文件、URL、作者、日期
- **一句话摘要**
- **该来源回答的问题**：必须从原文中提取一个独特的问题；**不要跨页面复用通用描述**
- **关键结构/章节线索**
- **值得保留的知识点**：必须是从原文提取的具体判断；**不要使用"文章中讨论 X 时可优先连接 Y"之类的通用指导**
- **关联的概念/实体/综合页**
- **后续精读任务**

### 实体页：`wiki/entities/`

实体页用于人、组织、产品、项目、工具等对象。实体页承载事实和背景，不应承载过长的论证。

- 仅为知识对象创建实体页：人、组织、产品、项目、工具等只有在它们本身是分析对象或承载可复用事实时才应进入实体层。不要因为某人是来源的作者就创建实体页。不要在实体页上维护文章作者列表。将作者信息保留在来源页的来源信息中或 `source/` 元数据中。

推荐结构：`定位 → 关键信息（结构化事实表）→ 核心能力/产品特性 → 在知识库中的角色 → 关联来源 → 关联页面`

### 概念页：`wiki/concepts/`

概念页用于方法、模式、理论、问题框架、框架等思想。优先复用和更新已有概念页，避免同义重复。

推荐结构：`定义 → 核心理解/判断 → 与邻近概念的关系 → 关联来源 → 关联页面`

**反模板规则**：

- 每个概念页的"与邻近概念的关系"部分必须逐一解释具体关系。**不要使用"它汇集了分散在多个来源中的观点，形成了一个可复用的概念"这类通用模板段落。**
- 每个概念页至少包含：清晰的定义、2 条以上独特的核心判断、与 2 个以上相关概念的关系说明。

### 综合页：`wiki/syntheses/`

综合页用于跨来源的判断、框架、主题概览、案例矩阵、架构分析等编纂知识。

每个综合页至少包含：核心判断、多来源综合（非单来源复述）、3 个以上来源页的交叉引用。

### 对比页：`wiki/comparisons/`

对比页用于区分易混淆的概念、产品、范式或解决方案，例如 `Reasoner vs Agent.md`。

推荐结构：

1. 一句话结论；
2. 被对比对象的定义；
3. 对比表格；
4. 决策方法/使用场景；
5. 常见误区；
6. 与知识库主线的关系；
7. 关联来源；
8. 关联页面。

最低质量要求：

- 对比表格的列名不能使用 `对象A / 对象B` 这类通用占位符，须写明具体的对比对象
- 必须说明"何时选 A，何时选 B"
- 至少链接到 2 个概念页或综合页
- 不要复用"在讨论模型能力时，需先判断……"这类适用于任何对比页的通用段落

### 主题图谱页：`wiki/syntheses/`

主题图谱页属于综合页的一种。它组织知识库的长期阅读路径和主题主线。不要放在 `Knowledge/` 根目录。

推荐位置：

- `wiki/syntheses/知识库主题图谱.md`
- 或按主题命名，如 `wiki/syntheses/Agent 工程主题图谱.md`

主题图谱应包含：

- 知识库当前的核心问题；
- 3-9 个主题主线；
- 每条主线的推荐阅读顺序；
- 入口概念页、关键综合页、核心来源页、相关对比页；
- 页面类型使用说明；
- 后续整理优先级。

与 `index.md` 的分工：

- `index.md` 维护完整的入口列表和页面目录；
- 主题图谱负责知识结构、阅读路径和主题导航；
- 不要让 `index.md` 承载过多说明内容，否则会变成过长的文章。

### 查询页：`wiki/queries/`

将有长期价值的问答输出归档为可复用的页面。每个查询页应回答一个明确的问题，并链接回相关的来源页、概念页和综合页。

## 核心工作流程

### 初始化知识库

1. 扫描 `source/` 下的目录、文件类型和来源分组；
2. 创建或更新 `wiki/index.md`、`wiki/log.md`、`wiki/README.md`；
3. 为首批材料创建来源页；
4. 提取实体、概念、综合和对比页面；
5. 更新索引和日志；
6. 检查断链、编码损坏、重复目录和空目录。

### 导入资料

1. 先阅读 `wiki/index.md`，定位已有的相关页面；
2. 阅读新来源，提取摘要、关键结构、实体、概念和可复用的判断；
3. 创建或更新来源页；
4. 更新已有的实体页、概念页和综合页。优先更新已有页面，不要轻易创建重复概念；
5. 标记关系：支持、补充、修正、矛盾或需验证；
6. 更新 `wiki/index.md` 和 `wiki/log.md`；
7. 执行断链和编码损坏检查。

### 批量导入主题

当一次导入同一主题的 5 篇以上资料时，不要仅创建多个来源页。还应建立主题结构。

步骤：

1. 判断是否已有对应的综合页或主题图谱条目；
2. 为每个来源创建来源页；
3. 创建或更新主题综合页，例如 `Claude Code 源码架构图谱`、`Agent 时代的创业方法论`；
4. 提取共享概念和实体，避免为每个来源创建重复概念；
5. 在 `index.md` 中以组形式呈现本次批量导入；
6. 将主题的阅读路径添加到主题图谱；
7. 按主题记录到 `log.md`，而非逐文件操作流水账。

### 基于知识库回答问题

1. 先阅读 `wiki/index.md`；
2. 然后读取相关页面。默认不要扫读整个知识库；
3. 在回答中引用 Wiki 页面链接；
4. 如果回答有长期价值，归档到 `wiki/queries/` 或更新综合页；
5. 更新 `wiki/log.md`。

### 知识库层面重组 / 主题重构

当 `source/` 已包含许多已导入材料，`wiki/` 页面数量增长，用户希望改善可导航性、主题聚类和长期可维护性时使用。

步骤：

1. 先运行审计脚本（或等效检查），确认：
   - 来源覆盖率
   - 断链
   - 索引覆盖率
   - UTF-8 问题
   - 薄弱页 / 模板残留
2. 阅读 `wiki/index.md`、`wiki/README.md`、`wiki/log.md` 和已有的综合页；
3. 识别知识库的主题主线，不要按来源目录机械组织；
4. 如果页面数量较多，创建或更新主题图谱页，例如 `wiki/syntheses/知识库主题图谱.md`；
5. 明确 `index.md` 和主题图谱的分工：
   - `index.md`：全量目录
   - 主题图谱：阅读路径、主题主线、知识架构
6. 重构优先级：
   - 薄弱的对比页
   - 空洞的概念页
   - 主题归属不清的综合页
   - `index.md` 中难以导航的长列表
7. 更新 `README.md`、`index.md`、`log.md`；
8. 最后重新运行断链、覆盖率、索引和编码检查。

### 充实已有页面

1. 阅读目标页面和相关来源；
2. 如果现有来源不足以支撑完整概念，可搜索外部信息并标注来源后补充；
3. 充实后更新页面的 `updated` 日期；
4. 检查双向链接完整性。

## 页面质量审计

定期检查薄弱页和模板残留。通过断链检查不代表知识质量达标。

### 薄弱页识别

应优先审查以下页面：

- 不足 800 字符的概念页
- 不足 800 字符的实体页
- 不足 1200 字符的综合页
- 不足 1000 字符的对比页
- 仅有摘要、缺少"该来源回答的问题"和"值得保留的知识点"的来源页

薄弱页不一定要扩充，但必须判断：

- 是否只是临时占位？
- 是否应合并到已有页面中？
- 是否应升级为完整概念页？
- 是否应在索引或主题图谱中降低优先级？

### 模板残留检查

特别查找以下内容：

- "它回答了某个长期问题"
- "如果文章讨论了"
- 通用的"为什么重要"章节
- "待补充"
- "TODO"
- `对象A / 对象B` 占位列名
- 通用的"如何使用本对比"段落

发现模板残留时，基于原始来源材料和相关页面将其改写为具体判断。不要仅删除标题。

## 审计脚本

可发布的审计脚本位于：

```bash
python3 Knowledge/schema/scripts/verify_wiki.py
```

为兼容已有命令，本知识库还在根目录保留了一个便捷包装：

```bash
python3 Knowledge/verify_wiki.py
```

发布技能时，将 `Knowledge/schema/scripts/verify_wiki.py` 作为 `scripts/verify_wiki.py` 与 `SKILL.md` 一起打包；根目录的包装仅为本知识库的便利入口点。

每次批量导入、重构、重命名或主题图谱更新后运行审计脚本。

通过标准：

- `Sources without ziliao: 0`
- `Broken wikilinks: 0`
- `Pages not in index.md: 0`
- `UTF-8 issues: 0`

如果脚本输出与人工判断冲突，以人工检查为准，但理由需记录在 `wiki/log.md` 中。

不要仅为通过脚本而机械创建低质量来源页。来源页仍需满足最低质量标准。

## `index.md` 规范

`wiki/index.md` 是导航入口。推荐结构：

```markdown
# 知识库索引

## 快速链接

## 来源

## 实体

## 概念

## 综合

## 对比

## 查询
```

每个记录最好包含：

```markdown
- [[Knowledge/wiki/concepts/上下文工程]] — 一句话描述；状态：演进中。
```

## `log.md` 规范

`wiki/log.md` 是基于日期的阶段变更时间线，不是过于细碎的操作流水账。更新原则：

- **按日期合并**：同一天执行的导入、重构、查询和维护尽量合并到 `## YYYY-MM-DD｜主题概述` 下，用 `###` 按主题分组
- **记录阶段成果**：保留关键来源、关键新创建或更新的页面、核心结论、重要维护决策。不要为每个小操作单独追加条目
- **必要时创建单独的详情页**：批量映射、长审计清单、图片迁移表、复杂实验日志等放在对应的综合页或 `wiki/queries/` 下。`log.md` 只链接到摘要
- **一天内多次更新时改写当天章节**：优先编辑合并当天已有章节，而非追加零散条目
- **保留后续方向**：可在末尾维护简短的"当前待办/后续方向"列表，但要保持简洁

推荐格式：

```markdown
## YYYY-MM-DD｜主题概述

### 主题一

- 导入/更新范围：……
- 创建/更新了：[[Knowledge/wiki/……]]、[[Knowledge/wiki/……]]
- 关键结论：……

### 主题二

- ……

## 当前待办 / 后续方向

- [ ] ……
```

常见主题：

- 初始化和结构调整
- 按主题导入来源
- 概念/综合页深化
- 查询归档
- 审计和维护
- 规范/技能规则更新

## 外部搜索边界

默认优先使用 `Knowledge/source/` 和 `Knowledge/wiki/` 下的内部材料。不要用外部搜索替代已有来源。

可在以下情况下使用外部搜索：

- 用户明确要求最新信息；
- 概念页需要标准定义或官方文档；
- 来源中的事实有时效性，可能影响结论；
- 需要验证产品、模型、公司、论文或法规的最新状态。

使用外部搜索时：

- 优先选择官方来源、论文和原始公告；
- 将外部来源链接添加到页面；
- 区分"原始来源的观点"和"外部验证的事实"；
- 不要将未经核实的网络信息混入稳定结论。

## 链接规则

- 使用 Obsidian Wiki 链接：`[[Knowledge/wiki/concepts/上下文工程]]`
- 链接到库内文件时，使用相对于库根的路径。不要使用绝对路径
- 重命名文件后，批量更新所有 `[[……]]` 链接
- 回复用户时，尽量优先使用可点击的 Wiki 链接
- **双向链接原则**：当概念 A 引用概念 B 时，B 的页面也应包含指向 A 的反向链接。边界模糊的概念对（如 Harness / Environment、Vibe Coding / Agentic coding）应在双方页面中说明其关系

## 编码安全规则

中文内容必须使用 UTF-8 编码。在 Windows / PowerShell 环境下，直接使用 PowerShell 的 here-string、`Add-Content` 或 `Set-Content` 编写中文很容易产生问号乱码。

推荐做法：

- 使用 Edit 工具修改 Markdown；
- 或创建 `.py` 脚本，用 `Path.write_text(..., encoding="utf-8")` 编写文本；
- 或用 Python 的 `read_text(..., encoding="utf-8")` 验证文件内容。

避免：

- 在 PowerShell 命令字符串中直接编写大段中文；
- 仅凭终端显示判断文件是否损坏。

每次批量写入后，检查：

- 重复的问号；
- Unicode 替换字符；
- 典型的乱码标记；
- 断链。

## 图片和附件

- 默认不要在 `source/` 中批量下载外部图片，除非用户要求或图片对长期理解至关重要
- 如果需要本地化图片，下载到库附件位置，并将 Markdown 图片语法转换为 Obsidian 图片嵌入：`![[image.png]]`

## 安全原则

- 不要覆盖用户的原始材料
- 不要删除非空目录，除非确认内容已迁移或用户明确要求删除
- 批量移动或重命名前创建映射。执行后检查断链
- 用户的显式偏好优先于本规范，例如目录命名方式、是否保留报告、`README.md` 是否保留英文名
