## 项目规则

- 请保持对话语言为中文，所有解释、回复均使用中文
- 该系统环境为 Windows，在提供命令、路径、脚本等建议时使用 Windows 风格
- 每次执行任务时候都分步执行，生成一个todo list
- 生成提交commit时候，用汉语生成

---

> 这些指南旨在减少常见的 LLM 编码错误。可根据需要与项目特定说明合并使用。
>
> **权衡取舍：** 这些指南偏向谨慎而非速度。对于简单任务，请自行判断。

## 1. 先思考，再编码

**不要假设。不要掩盖疑惑。暴露权衡。**

在实现之前：

- 明确陈述你的假设。如果不确定，请提问。
- 如果存在多种解释，请一并列出——不要默默自行选择。
- 如果有更简单的方案，请说出来。必要时提出异议。
- 如果有不清楚的地方，停下来。指出困惑之处。提问。

## 2. 简洁优先

**用最少的代码解决问题。不做推测性工作。**

- 不添加未被要求的功能。
- 不为一次性代码创建抽象。
- 不为未被要求的"灵活性"或"可配置性"做设计。
- 不为不可能发生的场景编写错误处理。
- 如果你写了 200 行但本可以用 50 行完成，请重写。

问自己："资深工程师会觉得这个过于复杂吗？"如果是，请简化。

## 3. 精准修改

**只动你必须要动的代码。只清理你自己造成的混乱。**

编辑现有代码时：

- 不要"改进"相邻的代码、注释或格式。
- 不要重构没有问题的部分。
- 遵循现有风格，即使你会有不同的写法。
- 如果你发现不相关的死代码，提一下——但不要删除它。

当你的改动产生了遗留代码：

- 移除因你的改动而变得无用的 import/变量/函数。
- 除非被要求，否则不要移除预先存在的死代码。

检验标准：每一行改动都应该能直接追溯到用户的需求。

## 4. 目标驱动执行

**定义成功标准。循环直到验证通过。**

将任务转化为可验证的目标：

- "添加验证" → "编写对无效输入的测试，然后让它们通过"
- "修复这个 bug" → "编写能复现它的测试，然后让它通过"
- "重构 X" → "确保重构前后测试都能通过"

对于多步骤任务，陈述简要计划：

```
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

明确的成功标准让你能独立推进。模糊的标准（"把它做好"）则需要不断澄清。

---

**这些指南有效的标志是：** diff 中不必要的改动减少，因过度复杂而重写的次数减少，澄清性问题出现在实现之前而非犯错之后。
